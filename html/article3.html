1How to design high-performing sites

Performance management step

All campaign and branding sites need to follow some core performance best practices. Performance needs to be at the centre of the architecture and a key consideration for all phases, including: requirements, design, development and testing.

The diagram below highlights the different steps involved in performance planning and management, and the sequence of these steps.


Capture performance requirements

This activity captures quantitative performance goals for different types of transactions (read only transactions, read-write transactions, transactions for a certain type of user (ex: admin user), for example). Use requirements templates that already factor-in performance and scalability requirements capture. Examples SLA Throughput Scalability.

Create performance model

Use models to document the latency in the system. Statistical modeling tools are useful here, but you could use any model that works. A simpler approach is to use a deployment or a component model along with performance related annotations (documenting the time taken in each component etc.) .Sequence diagrams are great for documenting intra-system latency. Prioritise the work flow and do it for the top few use cases. The workload model is derived from the captured requirements. From the requirements, we identify the performance scenarios. Each scenario constitutes a set of actions, and at least (but not limited to) three more things – throughput, response time and the load. These are then prioritised using the 80-20 rule, i.e. 20% of the functionality that is used 80% of time.

This workload model is used to create the test cases, environment, and benchmarks in those scenarios. This overall model that includes every other thing here now becomes a performance model. When run, this will identify the bottlenecks. It is very important to keep consulting this model with the actual users, as an incorrect model will result in having well-tuned software in areas that are rarely used.

Analyse performance models

Vary the performance models in various use cases to see how the system behaves in different situations. For example, see how the system behaves if the user has multiple accounts instead of one. Analyse the typical usage scenarios for the system, and use them to estimate the performance of the system.

Assess against requirements

Use the metrics and see if they solve the requirements. For example, is the SLA being met in the scaled-down system? What does an analysis of the performance model indicate?

Conduct practical testing

Use representative samples to conduct benchmark testing. Samples can be isolated or representative of the typical usage of the system. Scale the system and the sample in a similar way. Benchmark all Out of The Box (OOTB) applications. Use a recommended sample application if possible to benchmark OOTB parameters.

Rework the architecture

Consider optimising the architecture. Initiate all kinds of refactoring activities and temper the requirements to be more conducive to good performance. Provide alternatives to achieve the required goals.

2Performance factors

When you build for performance, there are two types of factors that can affect performance. These are primarily controlled factors and uncontrolled factors. Uncontrolled factors include elements like consumer’s device, internet speed, the operating systems and browsers on the consumer’s machine. You’ll not have much control over the management and optimisation of these, apart from ensuring the site is designed to work optimally under varying combinations. Controlled factors on the other hand are those where you have more control within the enterprise; HTML size, image sizes and resolution, javascripts, integration times, for example.

Page load times can be defined as the cumulative sum of:

  • Front end rendering times
  • Backend response time
  • Network traversal

Across each of these three components there are various controlled factors that need consideration while designing the solution.

The following sections highlight some of the key focus areas and best practices around the front end rendering times and backend response times.

3Front-end rendering optimisation

The front-end rendering time is one of the key factors affecting page response times, and is defined as the elapsed difference between the time when the first byte is received by the browser and the time when the complete page has been rendered (identified by the window load event).

The key factors to consider around front end rendering times are:

HTML – Images

Images form the most significant part of the Web Page, the number of images on the Page impacts the page render time:

  1. Use CSS spritesto combine your background images into a single image and use the CSS background-image and background-position properties to display the desired image segment.
  2. Allow parallel downloads for the images, this helps in downloading multiple images at once.
  3. Cache images on the browser
  4. Cache images on the AWS CDN network


Java Scripts & Stylesheets

Java scripts and stylesheets form the backbone of the client side validation, AJAX calls, and HTML layout. Often the number of java scripts and CSS will impact the page load.

  1. Combine the java scripts by common functionality into a single file and minify the java script: Product Details Scripts, Checkout Scripts, Search Scripts, for example.
  2. Load the java scripts only when needed. for example, a Java script that will be used after the entire HTML component is loaded should be put at the end.
  3. Combine CSS into a single CSS.
  4. Cache the java scripts and CSS on the AWS CDN network.
  5. Set expires date for all the Java scripts and CSS in future, this will allow them to be cached on the client browser.
  6. Load CSS in the HEAD section of HTML, this will allow the browser to display whatever content is available.


DOM

The higher the number of DOM elements, the slower the of DOM elements in the Java Script:

  1. During the HTML construction, ensure the number of DOMs are controlled.
  2. Carefully construct the HTML layout to use minimal DOM elements.

HTML Size

The amount of content shown on each page increases the size of the HTML. In most cases there is little room to reduce the size of the HTML page, but some of the options below can help:

  1. Have the components of HTML rendered only when it needs to be shown. For example, on PDP page, the latest review can be shown and when the guest selects ‘show more reviews’ the rest can be shown.
  2. Use AJAX to render content which is of less importance. For example, information about how the item can be returned and the shipping options availability on PDP page can be rendered after the complete page loads.
  3. Cache the HTML fragments wherever possible.

Number of DNS lookup

A DNS lookup takes approximately 20-120 ms of time to resolve the IPAddress of the server which will serve the content.

  1. Reduce the number of unique host names on the HTML page. It is best practice to keep the number of unique DNS to around four or five per site.

Parallel Downloads

The number of components (java script/CSS/HTML/Images) on the page impacts on the response time of the page. Most browsers can download two components (as per HTTP specification) per host name simultaneously, so if there is only one host rendering all the components of HTML, it would affect the page response:

  1. Allow parallel download by adding the multiple host names. Split the component to load from these host names. For example, on a page that has 20 components with four host names, eight components can be downloaded simultaneously.
  2. Use this approach only for bigger images, CSS, and Java script.
  3. Ensure that a particular component is always downloaded from a particular host name, otherwise the browser cache will not be of any use; i.e. all java scripts from a particular host name, and all CSS for one and so on.

Browser side Caching

  1. Leverage browser side caching for static assets.
  2. The assets should be cached with a reasonable TTL so that the requests don’t come to the app servers.
  3. We used a TTL of around 2 hours for all our static assets.


Information Architecture

The Information Architecture of a site can have a huge impact on its performance.

  1. Most CMS platforms have no hard limit on the number of items, or the depth of the content tree. However many browsers struggle with the time taken downloading the details of items with 100+ children.
  2. It is generally recommended to have: 12 wide, five deep as a rule of thumb. No more than 12 child items and no deeper than five levels.
  3. This gives you around 270k items, if the site is vastly more than this, it is recommended to use a taxonomy system.
  4. A taxonomy system can take many forms, but in essence it is where content is stored in a separate section to the website structure.
  5. The renderings pull information from content repository based on search criteria specified on the category nodes.
  6. You can enforce the depth limit in the CMS configurations.

4Backend response times

The Horizon platform provides a host of useful services that are available for all Diageo digital sites. These services provide core capabilities around age check, user profile management, registration, and notifications.
The performance of these services is managed by Infosys, however if you are planning to use the services as part of your digital platform ensure that:

  1. You have identified the services that you intend to use, and the anticipated volume that you will be injecting on to these services during peak times.
  2. The Infosys team that manages the services should confirm if they can handle the stipulated peak volume; or if they will have to do additional configuration, capacity or infrastructure changes to support your needs.
  3. In case of mid to high volumes (>10 TPS), coordinate synchronous performance testing with the team that manages the services to ensure performance benchmarking can be performed.
  4. Ensure that there is ancillary load also simulated on the services that would be generated from other digital platforms to simulate a real life situation.
  5. Ensure that you understand the timeouts imposed by the gateway for these services, and that it is factored into the user experience that you build for the digital platform.

It is important that you benchmark the page response times for your site in comparison to your competitors, but also ensure that you keep in mind the response times that your users would expect from your site.
